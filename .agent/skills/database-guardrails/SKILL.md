# Database Architecture & Guardrails Skill

This skill applies when you are modifying PostgreSQL schemas, writing Supabase RPCs, updating Row Level Security (RLS) policies, or dealing with backend database architecture.

## Database Schema (Supabase PostgreSQL)
* **SOURCE OF TRUTH:** You MUST read the `supabase_schema.sql` file in the root directory to understand the exact column names and types before writing any Supabase client queries or mutations.
* `platform_admins`: Global super-user directory protected by strict RLS preventing privilege escalation.
* `project_members`: Project-level relational table defining RBAC roles (`project_admin`, `gc_admin`, `design_team`, `viewer`). **NOTE:** The ENUM value is `project_admin`, NOT `owner` — using `owner` anywhere in code or queries will cause a silent type mismatch.
* `projects`: Master project records.
* `project_settings`: JSONB configurations for dynamic `building_areas`, `categories`, `sidebar_items`, `disciplines`, and `ve_column_order`. **`categories` shape:** `CategoryConfig[]` — `[{id: "...", label: "...", no_coord_default: false}]`. NEVER treat `categories` as a plain `string[]` — always normalize via `normalizeCategories()` from `lib/normalizeSettings.ts` to handle legacy string-array data gracefully at read-time. **`disciplines`, `permit_types`, `permit_ahjs`** follow the same `{id, label}` object shape with immutable UUID keys (see C7). **`ve_column_order` / `coord_column_order`** use `[{id: "...", label: "...", visible: true}]`.
* `opportunities`: The parent VE log items. Contains `design_markups` JSONB and `coordination_details` JSONB.
* `opportunity_options`: The relational contenders. Linked to `opportunities` via `opportunity_id` (Foreign Key, Cascade Delete). **DENORMALIZATION NOTE:** Also contains a `project_id` column explicitly for RLS performance, Realtime subscription scoping (`filter: project_id=eq.X`), and future cross-project global dashboard queries.
* **Loose Text Foreign Keys:** The platform uses raw string matching (e.g. `cost_code text`) rather than strict UUID Foreign Keys for global taxonomy items like `cost_codes`. This intentional denormalization enables cross-project analytics (`group by cost_code`) without requiring cascading deletes or cross-tenant dependency locking.
* **`cost_codes.is_division` Semantics:** The `is_division` flag is a **display/grouping hint for the viewer accordion ONLY**. It MUST NOT be used as a selectability filter in cost code dropdowns or comboboxes. All `cost_codes` rows — regardless of `is_division` value — are valid selectable cost codes. Adding `!c.is_division` or `.eq('is_division', false)` to any dropdown query is an anti-pattern that silently hides legitimate selectable codes (e.g. `260000 – Electrical` which is simultaneously a division header and a selectable cost code).
* `project_estimate_versions`: Versioned snapshots of a project's imported estimate. Each row holds a user-defined `version_name`, `version_date`, `total_budget` (pre-computed sum), and an `is_active` boolean. Only one version may be active per project at any time. The active version's `total_budget` drives `project_settings.original_budget` and the waterfall chart baseline.
* `project_estimates`: Line-item children of `project_estimate_versions`. Each row stores a `cost_code` (loose text FK to `cost_codes.code`), `cost_type` (plain `text` — NOT an enum; values: `'Labor' | 'Material' | 'Subcontract' | 'Equipment' | 'Other'`), `description`, `unit_qty`, `uom`, `unit_cost`, `budget_amount`, and `item_assumptions` (free-text scope notes). Cascade-deletes when its parent version is deleted. **Mutation constraint:** The table's RLS write policy is gated by `can_edit_project_settings`, which `design_team` users do NOT have. Edits to `item_assumptions` MUST go through the `update_estimate_assumptions` `SECURITY DEFINER` RPC — never a direct `.update()`. The RPC checks `can_edit_records` instead and batch-updates all lines matching a given `cost_code` within the active version. Using a direct `.update()` will silently return 0 rows affected with no error for `design_team` users.
* `estimate_variance_notes`: Stores user-provided explanations for cost swings captured during estimate upload. Granularity: `(estimate_version_id, cost_code)` — no cost_type drill-down. Protected by `enforce_variance_note_immutability` trigger that locks notes once the parent version is finalized (`is_finalized = true`). RLS scoped to project members via `project_id`. Fetched client-side via `useEstimateVarianceNotes()` hook in `useEstimateQueries.ts`, scoped to the active version only.
* `clients`: Global client directory for brand standards management. Contains `name`, `description`, `general_standards_url`, `primary_contact_name/email`, and soft-delete flags (`is_archived`, `is_deleted`). Linked to `projects` via `projects.client_id` (nullable FK, `ON DELETE SET NULL`).
* `client_brand_standards`: Global brand requirements defined per client. Uses loose-text `cost_code` and `normalized_csi_number` FKs. RLS scoped to platform admins + users with project access to the parent client.
* `project_brand_standards`: Project-level snapshots of global standards. Linked to `client_brand_standards` via `client_standard_id` FK (`ON DELETE SET NULL`). `UNIQUE(project_id, client_standard_id)` prevents duplicate syncs.
* `company_csi_defaults`: Global company-wide CSI-to-cost-code default mappings (the "Rosetta Stone"). Contains `csi_number` (UNIQUE constraint — 1:1 mapping per company), `description`, and `cost_code` (loose text FK to `cost_codes.code`, `ON DELETE SET NULL`). RLS: `SELECT` open to all authenticated users; `INSERT/UPDATE/DELETE` restricted to platform admins via `is_platform_admin()`. Mutation via `bulk_upsert_company_csi_defaults` RPC (SECURITY DEFINER, platform-admin-gated, set-based `jsonb_array_elements` insert with `ON CONFLICT (csi_number) DO UPDATE`). Seeding into projects via `seed_project_from_company_defaults` RPC which copies defaults into `project_csi_specs` with `source = 'company_default'` using `ON CONFLICT DO NOTHING` to preserve existing project-specific overrides.
* `project_csi_specs`: Project-level CSI specification mappings. Contains `source` column (`'company_default' | 'project' | 'ml_suggested'`) for lineage tracking. When a user modifies a seeded spec, the source transitions from `'company_default'` to `'project'` via upsert. The `trg_update_global_csi_training_data` trigger explicitly skips records with `source = 'company_default'` to prevent seeded data from polluting ML Flywheel training metrics. The Rosetta Stone aggregation RPC (`get_company_csi_rosetta_view`) filters to `source = 'project'` overrides only.
* `global_csi_training_data`: ML Flywheel training table storing CSI-to-cost-code mapping frequency data. Populated by `trg_update_global_csi_training_data` trigger on `project_csi_specs`. Protected by `is_default` guard — `company_default` source records are excluded from training. Cross-project aggregation available via `get_company_csi_rosetta_view` RPC.
* `project_sequences`: Stores sequence logic for project task ordering.
* `permits`, `permit_task_links`, `permit_comments`: Manages municipal permit applications, task associations, and commentary.
* `audit_logs`, `item_activity`: Centralized tables for tracking activity and granular changes across project items.
* `role_permissions`: Granular capability mappings extending the base user roles.
* `project_drawing_sets`, `project_sheets`, `sheet_markups`: Core schema for the Map Module. Stores uploaded PDF sheets, parsed vector markups, and drawing revisions.
* `client_documents`: Stores client-level attachments and standards documentation.
* `project_lessons`, `lesson_opportunity_links`, `lesson_attachments`: Manages lessons-learned capture across projects, linking them directly to VE opportunities and storing associated attachments.

## Backend Separation of Concerns
* **RLS & Junction Table Recursion:** When writing Row Level Security (RLS) policies for junction tables (like `project_members`) that check a user's role within that same table, NEVER query the table directly in the policy (e.g., `EXISTS (SELECT 1 FROM project_members...)`). This triggers infinite recursion and evaluation failures. You MUST create a `SECURITY DEFINER` helper function (like `get_user_project_role()`) to bypass RLS safely and use that function within the policy's `USING` or `WITH CHECK` clauses.
* **Foreign Keys to Auth:** When referencing `auth.users(id)`, ensure the constraint explicitly points to `auth.users` and that `REFERENCES` permissions are correctly established. RLS applies before foreign key checks, so ensure your RLS policies pass before diagnosing FK constraint errors.
* **Database Immutability & Escape Hatches:** When creating PostgreSQL triggers to enforce data immutability (e.g., locking financial records), always provide a transaction-scoped escape hatch (e.g., `IF current_setting('designpulse.bypass_immutability', true) = 'true' THEN RETURN NEW; END IF;`). This allows authorized, `SECURITY DEFINER` RPCs to intentionally unlock records without disabling the trigger globally. **CRITICAL TIMING:** In any RPC that writes to an immutability-protected table, you MUST call `PERFORM set_config('designpulse.bypass_immutability', 'true', true)` AFTER the RBAC permission check but BEFORE any `UPDATE` statement — including intermediate state resets (e.g., `SET status = 'Draft'`). Opening the hatch only near the final write will still trigger the immutability exception on earlier UPDATEs in the same transaction, causing a deadlock.
* **JSONB Type Casting & Safety in RPCs:** When iterating over JSONB objects in PL/pgSQL using `FOR k, v IN SELECT key, value FROM...`, strictly use `jsonb_each_text()` instead of `jsonb_each()` if you intend to compare the extracted values against standard SQL text/strings. Using `jsonb_each()` causes fatal `operator does not exist: text = jsonb` errors. Furthermore, ALWAYS wrap JSONB iteration loops in a type-safety check (e.g., `IF jsonb_typeof(COALESCE(column_name, '{}'::jsonb)) = 'object' THEN`) to prevent fatal transaction aborts if the frontend inadvertently saves a `NULL`, array `[]`, or scalar string to the column.
* **Anti-Polymorphic Relationships:** Strictly avoid polymorphic relationships for core security or activity tables. Always use explicit, nullable Foreign Key columns (e.g., `opportunity_id`, `option_id`) to ensure RLS policies can cleanly traverse relationships and `ON DELETE CASCADE` triggers function correctly.

## Code Generation Instructions
6.  **Relational Math:** Remember that an Opportunity's true financial weight prior to approval is derived from its attached Options, not just the parent row.
7.  **JSONB Data Integrity (Immutable IDs & Concurrent Safety):** When managing dynamic user-defined fields stored in `project_settings` JSONB arrays (`disciplines`, `categories`, `permit_types`, `permit_ahjs`, or any future settings array), always use immutable IDs (e.g., `crypto.randomUUID()`) as keys in the JSONB objects rather than display labels. If a user deletes a dynamic field, the data remains orphaned in the JSONB object. This acts as a "soft delete" data retention feature. Do NOT write cleanup scripts to delete orphaned JSONB keys. **CRITICAL:** When mutating heavily collaborative JSONB columns (like `coordination_details` and `coordination_requirements`), you MUST NOT rely on client-side state merging. Instead, the frontend must send strict delta payloads (only the fields that changed) to dedicated PostgreSQL RPCs. These RPCs must use `jsonb_set` or `||` coupled with `SELECT ... FOR UPDATE` row-locking to guarantee thread-safe atomic merges and prevent race conditions.
21. **JSONB Array Safety & Set-Based RPCs:** When processing bulk JSONB arrays in Supabase RPCs, you MUST validate the payload with `jsonb_typeof(p_payload) = 'array'` to prevent fatal transaction aborts on malformed objects. Furthermore, you must use high-performance set-based processing (`INSERT INTO ... SELECT ... FROM jsonb_array_elements(...)`) rather than iterative `FOR` loops to ensure atomic, instantaneous batch execution.
22. **Stable Grid Sorting (MVCC Tie-Breakers):** When fetching data for grids using non-unique columns like `created_at` (especially for bulk-imported data), you MUST include a deterministic tie-breaker in the `.order()` clause (e.g., `display_id` descending, then `id` ascending). Failure to do so causes PostgreSQL MVCC tuple replacements to alter the physical disk order during edits, resulting in fatal UI "row jumping" when real-time subscriptions refetch the query.
25. **Atomic Parent/Child Record Initialization:** Any `SECURITY DEFINER` RPC that creates a parent entity (e.g., `create_new_project`) MUST atomically initialize all required associated child records in the same transaction. Use `INSERT ... ON CONFLICT DO NOTHING` for idempotency, making the RPC safe to retry. Never assume that child records will be created lazily on first use — this creates a "silent hardcoded defaults" failure mode where the app appears to work but is actually ignoring user configuration. Example: `create_new_project` must immediately `INSERT INTO project_settings (project_id) VALUES (v_project.id) ON CONFLICT (project_id) DO NOTHING` before returning.
31. **Non-Destructive State Reversal (De-escalation Pattern):** When a record exists in multiple views simultaneously (e.g., a Coordination item that has been escalated into the Value Matrix), you MUST NEVER hard-delete it from one view — this destroys it in all views. Instead, implement a non-destructive "reversal" via a `SECURITY DEFINER` RPC. The canonical 4-step sequence is: **(1)** RBAC permission check, **(2)** open the `designpulse.bypass_immutability` escape hatch via `PERFORM set_config('designpulse.bypass_immutability', 'true', true)` — a single transaction-scoped call covers ALL tables with immutability triggers touched in this transaction (both `opportunities` and `opportunity_options`), **(3)** unlock child records (e.g., `opportunity_options`) FIRST — `trg_sync_parent_opportunity_totals` fires here and recalculates the parent financials from now-unlocked children; the explicit parent reset in step 4 then takes final precedence, **(4)** reset the parent record's financial fields to zero and strip any view-routing flags from the `coordination_details` JSONB using the `-` key-deletion operator with a `COALESCE` null guard (e.g., `COALESCE(coordination_details, '{}'::jsonb) - 'is_escalated'`). The UI MUST replace any generic "Delete" button on a shared record with a context-aware "Remove from [View]" action that calls the RPC. The record survives intact in its origin view — only the routing flag is removed.
33. **Estimate Version Activation is Always an Atomic RPC:** The operation of setting a project estimate version as "active" MUST be performed exclusively via the `activate_estimate_version` `SECURITY DEFINER` RPC. Never implement this as two sequential client-side mutations (e.g., first deactivate the old version, then activate the new one) — a network failure between calls leaves the project with no active version or two active versions simultaneously, breaking `project_settings.original_budget` sync and the waterfall chart. The RPC enforces the following in a single atomic transaction: **(1)** RBAC check via `has_project_permission(project_id, 'can_edit_project_settings')`, **(2)** `UPDATE ... SET is_active = false WHERE project_id = X AND is_active = true` to deactivate all current versions, **(3)** `UPDATE ... SET is_active = true WHERE id = p_version_id` to activate the target, **(4)** `SELECT ... FOR UPDATE` row-lock on `project_settings` to prevent concurrent sync races, **(5)** `UPDATE project_settings SET original_budget = v_total` with the activated version's pre-computed `total_budget`. The TanStack Query mutation's `onSuccess` handler must invalidate both `['estimate-versions', projectId]` and `['project-settings', projectId]` to keep the Project Info tab's budget field in sync without a page reload.
